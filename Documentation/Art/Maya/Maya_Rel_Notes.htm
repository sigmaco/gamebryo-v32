Gamebryo LightSpeed 3.2

  • Removed support for Maya 8.5

    • The creation of static actors is now supported.

  • Fixed an issue that prevented exporting scenes that used used hardware instancing with custom shaders.
  • Fixed an issue that caused the NiMultiShader interface to disappear when working with a file that references a custom shader that no longer exists.

Gamebryo LightSpeed 3.1

  • Added support for Maya 2010
  • Improvements were made to the NiMultiShader attribute editor interface. These improvements were designed to solve usability problems.   • Many improvements were made to the Maya exporter options. The significant one is now options are loaded when the scene is loaded. Prior versions would only load the options when you either exported or opened the option dialog.
  • File export path is now stored as a relative path to the scene.
  • Added a new feature to have the exporter completely ignore nodes under a "GamebryoIgnore" dag node in the scene graph. This is useful for having nodes that are needed for art creation but not really needed at time for export.
  • Gamebryo now supports D3D11 within the Maya plug-in. It is now possible to export textures specifically for D3D11, and a D3D11 viewer can be launched within Maya.
  • Added export option to control whether external original textures are copied to the export location.
  • The ExportLog.xml file is now properly named and formatted when the silent export option is set.
  • The Maya plug-in now identifies and reports more export errors rather than indicating success when failures occur.
  • The Maya plug-in now exports the correct skin bind pose when multiple smooth skins are applied to the same joints.

Gamebryo 2.6.0

  • Added support for Maya 2009
  • Support added for specifying Anisotropic Filtering as a filter mode in the texture export custom attributes. For more information, see the NiRenderer documentation and the documentation for each platform-specific renderer.

The behavior is unchanged when "Convert PSys to Fluid" is unchecked in the PhysX Extractor options. You can use this feature to create static actors (although it won't work if you have particles in the scene that you don't want converted to fluids).

Known Issues and Limitations

All Releases

PhysX

All Releases

  • Selecting the Shade Group as the current texture in the UV Texture Editor when using the NiMultiShader may cause Maya to crash.
  • Changing a material to use the standard material from a custom shader, and then re-assigning the original custom shader may cause the NiMultiShader to fail to generate the custom shader's parameter UI. If this happens, select a different custom shader, then return to the original to re-generate the UI.
  • Export of Bone / Skin LOD requires the SceneGraphOptimization plug-in.
  • Maya cannot open any scenes containing an NiMultiShader in batch mode. For this reason, it is recommended to launch Maya in GUI mode even when doing automated batch processing.
  • Maya may crash when pressing a button on the Particle Shelf without having the Gamebryo Maya Plug-in loaded. To avoid this problem preload the Gamebryo Maya Plug-in in the standard Maya plug-in manager.
  • Maya 2009 may crash at shutdown on quad core PCs using the Gamebryo Maya Plug-in. This problem is annoying, but non-destructive since the scene is already saved or discarded by the time the crash occurs.
  • MultiNif exporting may not work properly unless a FileName is explicitly specified for each part to be exported in the MultiNif Export dialog. This is true even if the global or local settings specify to use the object name as the name of the NIF.
  • The first time the plugin is loaded custom viewer scripts are generated for each viewer plugin, such as the Xbox 360 and PS3 viewer plugin. However, these viewer scripts are not available for use until the second time the plugin is loaded.
  • When the Gamebryo plug-in is loaded on demand, in response to a file load, an extra export parameters node will be created that takes precendence over the parameters specified in the loaded file. Always explicitly load, or auto-load, the Gamebryo plug-in before loading a file that needs it.
  • Export NonVisible Objects (as culled) option does not do anything. Non-visible objects are aways exported as culled.
  • Parsing mesh profiles can take a significant amount of time. The more files needed to parse will increase the time it takes Maya to load the plug-in. The time is directly proportional to the number of profiles. The number of profiles should be monitored and kept to a minimum.
  • With certain hierarchies the exporter fails to sample animation properly. To work around this issue, turn off sampling and bake the animation inside Maya.
  • If the exporter has to create generated nodes, for instance in the case of a transform with a non-identity pivot, if there is a character root attribute, it will not be placed on the correct generated node. This will cause the controller extractor to potentially miss animation on the character root. This might also apply to other attributes, besides the character root attribute.
  • The Maya compression dialog has issues when setting values and occasionally does not reflect the values that are actually exported.
  • Upon manual deletion of one of the maps from the NiMultiShader, the last known component values will persist. This may cause the NiMultiShader to shade incorrectly. For example, if when manually deleting the "DarkMap" from the NiMultiShader, the components could be left in an all dark state (0, 0, 0). In this situation, the object may render black inside maya. The work around for this side effect is to use Maya's "Attribute Spread Sheet" and set the DarkMap's R,G,B values back to the default of (1,1,1).
  • JPEG files are not converted when used as environment maps. A native format must be used.
  • Flipbook textures will not be compressed by the Gamebryo Maya Plug-in even if compression is specified. If compressed textures are needed, manually compress the textures.
  • A dialog may be displayed that says "The procedure entry point cgIsParameterUsed could not be located in the dynamic link library cg.dll.", because Maya's older version of Cg is being used instead of NVIDIA's. The solution is to edit the system path environment variable to remove Maya's bin directory. This directory contains an older version of Cg that should not be used with Gamebryo.
  • Particles will export to Gamebryo as double the size of their viewport sprite representations.
  • Procedural textures may export with the wrong repeat value for their UVs.
  • Coverage, Translate Frame, and Rotate Frame of place2dtexture would occasionally export in prior releases and this functionality has been removed. If you have any issues converting art assets for the new behavior, please contact support.
  • Per texture size reduction fails for HDR images when exported for Xenon. Either scale down HDR images with global exporter settings, or modify the source images.
  • The accumulation root for a character should not be used as a bone for skinning any vertices. This is because some of the transformations on that bone will be propagated to a NonAccum node, and the skinning will no longer look correct. This will be addressed in the future, but in the meantime do not use the accumulation root for skinning.
  • NiEulerRotKeys have limitations in how they are used for rotation accumulation. If any axis is specified for rotation accumulation, then all axes will be accumulated.
  • If a controller has keys that are looped in the source art package and the sequence time range does not include those keys, a single clamped value will be computed from the keys and used for that sequence, eliminating the looping behavior. A workaround for this is to manually loop the animation keyframes within the sequence time range. This problem also affects IFL files that are used with sequences. In the case of an IFL file, a workaround is to manually loop the listed frames in the IFL file itself so that the animation loops throughout the entire time range.
  • Textures other than TGA, BMP, SGI, RGB, RGBA, INT, INTA, DDS are occasionally smeared. This is a known Maya texture export issue.
  • The right-click menu for animation controls in the Attribute Editor does not work with NiMultiShader's Material Alpha parameter.
  • Use of non-uniform scale on bones must be avoided. Gamebryo only supports a uniform scale as a trade-off of flexibility for performance. Unlike the general case, it is not possible to propagate non-uniform scaling information directly to the vertices.
  • Only the XYZ rotation order is supported, if Euler rotations are used.
  • Unsigned Integer shader values are stored as signed integer values in Maya. This is a limitation of Maya.
  • Shaders that are provided as FX files will not have their pixel and vertex shader versions correctly reported. They will always display shader version 0.0.
  • Tangents weights are not supported, and different tangent values for position keys are not supported. If your animation utilizes either one of these qualities, the animation will need to be sampled. Animations are sampled by default.
  • Sometimes the export progress window will remain up even though the export has finished. When this happens just close the window.
  • Scriptjobs are not run in batch mode. This is a limitation of Maya, however, certain Gamebryo scripts are run using Scriptjobs that update the Maya scenes. These will not run in batch mode. To work around the issue, open older scenes in Interactive mode and then re-save them. This is probably not an issue for most scenes, but will affect scenes with 4 bit texture options.
  • The light Attribute Editor can get in a state where it will not view Gamebryo Attributes. If two different types of lights get Gamebryo Attributes added to them, only the last type of light will be able to display the attributes. The work around requires that the user call the MEL command "resetAE."
  • When using the Texture Set controls in the NiMultiShader interface if the uv sets are specified before the textures are added to the shader the values are ignored.
  • Maya 2008 produces non-unitized surface normals on some computers (for example, 0, 0, 0.99999994). During export, the Gamebryo plug-in will attempt to re-normalize. Due to precision errors, this will still result in non-unitized normals (e.g. 0, 0, 1.001959).
  • When exporting blend shapes for the PS3, keep the number of targets for the blend below 20.
  • The default particle system colliders created by the Gamebryo toolbar in Maya will be set to tell particles to spawn on collision but not die on collision, but the spawner applied to those particle systems is not set to spawn any particles. This will have the same overall effect as not spawning at all, but it may introduce some additional unnecessary work.
  • The Xbox 360 does not support 24-bit images. To avoid runtime image conversion and possible memory fragmentation, 24-bit images will be converted to 32 bits during Xbox 360-specific export.
  • The Playstation3 does not support 24-bit images. To avoid runtime image conversion and possible memory fragmentation, 24-bit images will be converted to 32 bits during Playstation3-specific export.
  • The Maya plug-in may not correctly convert rigid skins when multiple meshes have been applied to the same joint.

PhysX

  • The Maya PhysX plug-in is available for Maya 2009 and Maya 2010. See NVIDIA PhysX™ Resources for additional information about PhysX.
  • Cannot create joints to fixed world locations—Any rigid constraint (joint) must be connected to two rigid bodies when authored in Maya. However, PhysX requires joints to static locations to be specified with no actor and a pure world location.

    Workaround: Create the joint to a static actor in the scene. Typically, you should make the static actor the second rigid body for the joint. Check the "Same Position" option. Upon export, the joint will replace the static actor with the pure world location and behave as expected. This joint will not be created when you simulate in Maya due to bugs in the PhysX Maya plug-in.

  • Instancing of PhysX not Supported—The PhysX Maya plug-in does not support instancing of PhysX content.

    Workaround: Instance first, add PhysX properties after.

  • Duplication of PhysX content fails—The PhysX Maya plug-in crashes when you try to duplicate PhysX nodes.

    Workaround: Don't try to copy or duplicate. Make copies of the original geometry and then add PhysX properties. For very large scenes it may be worth scripting the process.

  • Rigid Bodies lose Lighting—When an object is converted to a rigid body the light links are lost.

    Workaround: Re-link the lights explicitly.

  • Scale on Accumulation Root not Supported—Scaling transformations of any kind on the "Accumulation Root" of a character will cause physics control of the character to fail.

    Workaround: Create an intermediate node as a child of the Accumulation Root and scale it instead of the root.

  • Maya Gravity is Ignored—The Gamebryo export process always uses the gravity vector given in the PhysX Extractor tool plug-in options.

    Workaround: Make certain that the PhysX Extractor gravity vector matches the gravity you are using internally for Maya.

  • Fluid Emitters Cannot be Attached to Triangle Mesh Shapes—A fluid emitter cannot be attached to an actor that has a triangle mesh shape.

    Workaround:Use an alternate collision shape for the actor.
