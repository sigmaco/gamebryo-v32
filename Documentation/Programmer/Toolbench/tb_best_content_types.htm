Toolbench Plug-in Best Practice: Content Types

The Toolbench content system is central to how data is stored and presented to users. It is important to give careful thought to what file types should rate as top level content. Top level content is typically derived from AbstractDocument and has its own undo/redo stack. It can be created through the New Content dialog, it is easily accessible from the Game Solution view (or Behaviors view), and it opens in its own document tab. An EMT project is an example of top level content. If, however, an EMT model was considered top level content, there would be an explosion of open document tabs, far too many separate undo stacks, and other complications.

New content types can indicate whether or not they require attachment to a game solution. The game solution is responsible for establishing the active asset web via Asset Controller, so this may factor into the decision. Content types should have a unique file extension. The .xml extension should not be used for a new Toolbench content type. If the content type requires a game solution, it should not have a file association with Toolbench and will not appear in the Most Recently Used (MRU) files menu.
