Best practices for working with entity properties

Emergent recommends the following best practices when you are working with entity properties:

  • Use property types that are appropriate for the type of data you're storing. Although it can be convenient to store all of your properties as strings, avoid this temptation. You should use types that are appropriate for the type of data you are storing. This helps avoid unnecessary conversions. To print values in Lua, use the tostring or string.format Lua functions.
  • Use the 'ReadOnly' trait for data that is set on the Model and never specialized for an Entity. For data that won't change at runtime, the 'ReadOnly' trait should be set. This prevents accidental updates of the property value at runtime. It can save space because properties are managed as copy-on-write, so ReadOnly properties exist only in the Model description in memory. These properties cannot be edited in World Builder.
  • Avoid exposing server properties to clients by using the 'ServerOnly' trait. By adding the 'ServerOnly' trait to your properties, the properties become visible only to server processes. 'ServerOnly' properties are available only to your server for processing and are not even seen by clients. This can be useful for security and fair-play reasons.
  • Use the 'Hidden' trait to hide properties that are defined in the Entity Modeling Tool from being edited in World Builder. World Builder provides a powerful way to override default property values for individual entities. There may be times when you do not want users overriding your EMT-specified default values for Entity instances in World Builder. For these situations, set the 'Hidden' trait on the properties. These properties are still associated with entities and their default values remain consistent when placed in the world. Non 'ReadOnly' 'Hidden' properties can still be updated at runtime.
  • Name Properties as nouns and use descriptive names.
  • Be aware that Property names are unique across the game title. This is so that any mixing can properly resolve Properties that having the same name. If you need Properties in different model hierarchies to remain independent, consider using a differentiating prefix.
  • Prefer using EntityID values instead of UInt64 values to represent Entities in properties. This allows the Lua runtime to automatically convert these properties to Lua Entity proxies. Using UInt64 values requires your to call Emergent:GetEntity to obtain the proxy object for the entity.
