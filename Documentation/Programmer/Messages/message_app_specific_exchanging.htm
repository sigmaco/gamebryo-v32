Exchanging application-specific messages between applications that are located on separate hosts

efd::NetService is a platform independent remote communication abstraction, available in the Foundation libraries. The Network System is optional, but if it is present the Message System will use it to automatically route messages remotely.

The Network System uses Categories to determine how to route messages. All messages sent through the Message System are sent on a Category which the Network System uses to determine which remote processes to route the message to. When the Message System subscribes to a Category the Network System is notified and automatically updates routing information in remote processes to ensure remote messages with that Category are forwarded to the subscriber.

Messages sent using the Network System can use different qualities of service which determine the type of underlying transport used. New qualities of service can be plugged in depending on application needs. Examples of qualities of service that might be useful include compressed, encrypted, data rate limited, sending only the most recent update, and any application specific way of determining priority or order of messages to send.

The Network System is only supported for tool time use but is designed for general purpose application.

Exchanging messages between separate hosts or processes on the same host is actually very similar to sending local messages. All of the same setup for creating message class ids, message classes, handlers, and subscriptions is the same. The only exception is that a Category other that kCAT_LocalMessage must be used. The Category kCAT_LocalMessage can only be used for messages sent to the local process.

To exchange an application-specific message between application that are located on separate hosts

  1. Create an instance of NetService in both processes that you wish to communicate between.
  2. Create a single instance of ChannelManagerService in one of the processes or in a separate process.
  3. Modify the config file (or pass the appropriate command line parameters) for each process to point to the IP address and port that the ChannelManagerService is running on.
  4. Send a message on a Category other than kCAT_LocalMessage. If the remote process is subscribed to that Category, the message should be delivered.
